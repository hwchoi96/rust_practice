# Cursor / AI 협업 규칙

이 파일의 **주된 목적**은 **사용자와 AI가 이 레포에서 일할 때 지킬 약속**을 적어 두는 것이다.  
프로젝트 구조, 학습 주제, 바이너리·데이터 파일, 대용량 파일 실습 등 **도메인 내용·요약은 `readme.md`**에 둔다. (여기서는 반복하지 않는다.)

## Cursor / AI와의 약속

- 사용자가 **코드를 레포에 직접 써 달라고 하지 않고** “알려줘”, “설명해”, “너가 짜지 말고”처럼 말한 경우: **답은 대화(채팅)로만** 하고, **`src/` 등 소스 파일을 대신 수정하지 않는다.** (직접 타이핑하며 배우는 흐름을 존중한다.)
- 반대로 “구현해”, “파일에 반영해”, “수정해”처럼 **저장까지 위임**하면 그때는 레포에 편집해도 된다.

## 코드 리뷰 시 (AI가 따를 규칙)

사용자가 **코드 리뷰**를 요청했을 때 아래를 **빠짐없이** 반영한다. (목표: **고성능·저지연** Rust와 **안전한 동시성**에 맞는 피드백.)

1. **잘된 점 / 아쉬운 점**  
   - 둘 다 **구체적으로 지적**한다.  
   - 각 항목마다 **왜 그렇게 판단했는지 사유**를 짧게 덧붙인다 (러스트 관용·API 계약·메모리·가독성 등).

2. **예외·오류 처리**  
   - `Result` / `Option` / 패닉 가능 지점 / I/O·파싱 실패 등 **예외적 상황**이 코드에서 어떻게 다뤄지는지 본다.  
   - 부족하면 **어느 위치**를 **어떻게** 바꾸면 좋은지 서술하고, 가능하면 **짧은 샘플 코드**로 보여준다.

3. **성능·지연시간**  
   - 사용자는 **고성능·저지연** 프로그램을 목표로 한다.  
   - 불리한 패턴(불필요한 할당, 복사, 잠금, 시스템 콜 빈도, 버퍼 크기, `println!` 남용 등)이 있으면 **지적**하고, **대안**을 제시한다.  
   - 대안은 **표준 라이브러리만**인 경우와 **`Cargo.toml`에 의존성을 추가하는 경우**를 **둘 다** 가능하면 나열하고, 의존성을 쓸 때는 **크레이트 이름과 역할**을 함께 적는다.  
   - 필요 시 **샘플 코드**로 비교할 수 있게 한다.

4. **동시성**  
   - 성능 이야기와 함께 **데이터 레이스·데드락·공유 가변 상태·`Send`/`Sync`·채널·`Arc<Mutex<_>>` 남용** 등 **동시성 이슈가 없는지** 검토한다.  
   - 단일 스레드 코드라도 “나중에 스레드를 늘릴 때” 걸리는 점이 있으면 짚는다.
